프로젝트 설명

MonsterWolrd

3D 오픈 월드 환경에서 몬스터를 포획하고, 포획한 몬스터를 아군으로 소환해 함께 전투하는 1인 액션 RPG 프로젝트입니다. 총·활·볼의 세 가지 전투 수단을 상황에 따라 전환할 수 있도록 구성했고, 몬스터 AI를 상태머신으로 설계해 적/아군 전환이 가능한 구조를 구현했습니다.

개발기간 2025.01 ~ 2025.03
장르 3D 액션 RPG
개발 형태 1인 개발
핵심 구조 상태머신 기반 AI / 오브젝트 풀 / 데이터 테이블

이 프로젝트의 목표는 단순히 전투가 되는 게임을 만드는 것이 아니라, 포획 전후에 몬스터의 행동이 자연스럽게 전환되는 구조와 다양한 전투 요소를 안정적으로 관리할 수 있는 설계 방식을 검증하는 것이었습니다. 이를 위해 상태머신, 오브젝트 풀, 데이터 테이블, 세이브 시스템을 각각 독립적인 구조로 분리해 기능이 늘어나도 유지보수가 가능한 형태를 만드는 데 집중했습니다.

설계 포인트

제네릭 상태머신 기반 몬스터 AI

몬스터 행동을 Idle, Patrol, Chase, Attack, Return 상태로 구분하고, 각 상태의 Enter / Update / Exit 동작을 Dictionary 기반으로 등록하는 구조를 사용했습니다. 이를 통해 switch 문이 비대해지는 문제를 줄였고, 몬스터별 AI 확장도 비교적 단순하게 처리할 수 있었습니다.

적/아군 전환형 단일 몬스터 구조

포획 전에는 적 AI로, 포획 후에는 아군 AI로 동작해야 했기 때문에 몬스터 클래스를 두 벌로 나누지 않고 하나의 클래스 안에서 friendStateEnter / friendStateUpdate 구조를 별도로 구성했습니다. 같은 데이터 구조를 유지하면서도 행동 패턴만 바꾸는 방식으로 설계 복잡도를 줄였습니다.

오브젝트 풀 + 데이터 테이블 구조

투사체, 스킬, 이펙트는 Queue 기반 오브젝트 풀로 재사용되도록 만들었고, 몬스터/레벨/스킬 수치는 CSV 데이터와 JSON 저장 구조로 분리했습니다. 덕분에 전투 연출과 밸런스 수치를 코드 수정 없이 조정할 수 있는 기반을 만들 수 있었습니다.

문제 해결

문제 1. 몬스터 AI 분기 증가

몬스터 상태가 늘어날수록 if / switch 기반 분기가 빠르게 복잡해졌습니다. 이를 해결하기 위해 상태별 함수를 Dictionary에 등록하는 제네릭 상태머신 구조를 사용했고, 상태 추가 시 기존 흐름을 크게 건드리지 않도록 정리했습니다.

문제 2. 포획 후 아군 전환 구조

포획 전과 후의 행동 패턴이 완전히 달라 별도 클래스로 분리할 수도 있었지만, 공통 데이터와 애니메이션 처리 중복이 커질 수 있었습니다. 그래서 하나의 Monster 클래스 내부에 적군용 상태와 아군용 상태 딕셔너리를 따로 두어 구조 중복을 줄였습니다.

문제 3. 전투 중 생성/삭제로 인한 비용 증가

투사체와 스킬 이펙트가 반복 생성되면 GC 부담과 프레임 저하가 발생할 수 있어, Queue 기반 오브젝트 풀을 직접 구현해 재사용하도록 구성했습니다. 이 방식으로 전투 반복 상황에서도 비교적 안정적인 흐름을 유지할 수 있었습니다.

핵심 코드

제네릭 상태머신 구조

몬스터 AI와 같은 반복적인 상태 전환 로직을 한 클래스에서 관리하기 위해 제네릭 상태머신을 만들었습니다. 상태별 Enter / Update / Exit를 각각 Dictionary에 등록해, 상태가 추가되더라도 흐름을 쉽게 확장할 수 있도록 설계했습니다. 

using System.Collections.Generic;
using UnityEngine;

public class StateMachineBase : MonoBehaviour
{
    protected delegate void StateFunction();

    protected Dictionary _stateEnter;
    protected Dictionary _stateUpdates;
    protected Dictionary _stateExit;

    public T _currentState;
    protected T _prevState;

    protected void InitStateMachine()
    {
        _stateEnter = new Dictionary();
        _stateUpdates = new Dictionary();
        _stateExit = new Dictionary();
    }

    protected virtual void ChangeState(T state)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_stateExit.ContainsKey(_prevState))
            _stateExit[_prevState]();

        if (_stateEnter.ContainsKey(_currentState))
            _stateEnter[_currentState]();
    }
}

적/아군 전환형 몬스터 AI

하나의 Monster 클래스 안에서 적군과 아군의 상태 흐름을 따로 구성해, 포획 전후 행동 패턴을 분리했습니다. 공통 데이터는 유지하면서도 실제 행동 규칙만 바꾸는 구조라서 클래스 중복을 줄일 수 있었습니다. 

protected Dictionary _friendStateEnter;
protected Dictionary _friendStateUpdate;
protected Dictionary _friendStateExit;

protected override void ChangeState(MonsterState state)
{
    _checkTime = 0;
    _stateTime = 0;

    if (_isFriend)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_friendStateExit.ContainsKey(_prevState))
            _friendStateExit[_prevState]();

        if (_friendStateEnter.ContainsKey(_currentState))
            _friendStateEnter[_currentState]();

        return;
    }
    base.ChangeState(state);
}

JSON 기반 세이브 데이터 관리

세이브 슬롯을 Dictionary로 관리하고, 실제 저장은 JsonUtility와 파일 입출력으로 처리했습니다. 플레이어 위치, 레벨, 포획 몬스터 정보를 구조체로 보관해 타이틀 씬과 인게임 씬에서 공통으로 사용할 수 있는 형태로 구성했습니다. 

public void LoadAllData()
{
    _slotData = new Dictionary();

    int slotCount = (int)Slot.Max;

    for (int i = 0; i < slotCount; i++)
    {
        Slot slot = (Slot)i;
        string fileName = (slot).ToString();

        if (File.Exists(_path + fileName))
        {
            string JsonData = File.ReadAllText(_path + fileName);
            PlayerData data = JsonUtility.FromJson(JsonData);
            _slotData.Add(slot, data);
        }
        else
            _slotData.Add(slot, _newData);
    }
}

배운 점

이 프로젝트를 통해 AI를 단순히 조건문으로 구현하는 것보다, 상태 단위로 책임을 분리해 설계하는 것이 훨씬 유지보수에 유리하다는 점을 체감했습니다. 특히 포획 전후처럼 행동 패턴이 크게 바뀌는 경우에도 공통 구조를 유지하면서 확장할 수 있는 설계가 중요하다는 점을 배웠습니다.

개선 방향

앞으로는 전투와 AI를 더 데이터 중심으로 분리하고, 몬스터 종류별 고유 패턴을 ScriptableObject 또는 별도 행동 트리 구조로 확장해볼 계획입니다. 또한 현재의 매니저 중심 구조를 좀 더 느슨한 의존성 구조로 바꿔 테스트와 유지보수를 쉽게 만드는 방향도 검토하고 있습니다.
