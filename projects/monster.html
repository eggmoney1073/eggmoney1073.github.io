Overview

프로젝트 개요

무엇을 만들었는가

플레이어는 전투와 포획을 통해 몬스터를 수집하고 성장합니다. 몬스터는 적군과 아군 양쪽 역할을 수행할 수 있도록 설계되었고, 각 몬스터는 상태에 따라 순찰, 추적, 공격, 복귀를 반복합니다.

왜 중요한가

저는 재미있는 플레이 경험을 오래 유지하려면 조작과 전투 흐름뿐 아니라 내부 구조도 탄탄해야 한다고 생각합니다. 그래서 몬스터 행동, 스킬 실행, 저장 데이터, 테이블 데이터를 분리해 성능 관리와 기능 확장 모두에 유리한 형태로 설계했습니다.

Architecture

핵심 구조 분석

1. 제네릭 상태머신 베이스 클래스

상태 전환 구조를 Monster 전용으로 하드코딩하지 않고, 제네릭 기반 상태머신 베이스로 만들어 재사용 가능한 형태로 구성했습니다. Enter, Update, Exit를 Dictionary로 관리하는 방식입니다.

2. 몬스터 하나로 적군과 아군을 모두 처리

Monster 클래스 내부에서 _isFriend 기준으로 상태 전환과 업데이트 흐름을 분기합니다. 즉, 별도 FriendMonster 클래스를 만드는 대신 하나의 구조 안에서 전투 역할을 전환합니다.

3. SkillManager와 풀링 구조

스킬은 몬스터가 직접 생성하지 않고 SkillManager에 요청합니다. SkillType별 풀을 관리해 스킬 오브젝트를 재사용하고, 몬스터는 어떤 스킬을 언제 사용할지만 결정합니다.

4. JSON 기반 저장 데이터 관리

슬롯 데이터는 DataManager가 전담합니다. 현재 슬롯과 현재 데이터를 분리하고, JsonUtility로 직렬화해 저장하는 구조라 세이브와 로드 흐름이 비교적 명확합니다.

State Machine Base

재사용 가능한 상태머신 베이스

상태별 Enter, Update, Exit를 Dictionary로 관리한다. 이 구조 덕분에 Monster와 Skill 같은 여러 객체에 같은 패턴을 재사용할 수 있다. 

public class StateMachineBase<T> : MonoBehaviour
{
    protected delegate void StateFunction();

    protected Dictionary<T, StateFunction> _stateEnter;
    protected Dictionary<T, StateFunction> _stateUpdates;
    protected Dictionary<T, StateFunction> _stateExit;

    protected virtual void ChangeState(T state)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_stateExit.ContainsKey(_prevState))
            _stateExit[_prevState]();

        if (_stateEnter.ContainsKey(_currentState))
            _stateEnter[_currentState]();
    }
}

몬스터 데이터를 테이블 기반으로 초기화

몬스터 타입과 레벨에 맞는 데이터를 Manager에서 받아와 체력, 공격력 같은 실제 전투 수치를 계산한다. 즉, 몬스터 동작 코드와 수치 데이터가 분리되어 있다.

Monster AI

적군과 아군을 하나의 클래스에서 분기

_isFriend 여부에 따라 상태 전환과 업데이트 흐름을 다르게 사용한다. 구조를 통합해 중복 클래스를 줄인 점이 특징이다. 

protected override void ChangeState(MonsterState state)
{
    _checkTime = 0;
    _stateTime = 0;

    if (_isFriend)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_friendStateExit.ContainsKey(_prevState))
            _friendStateExit[_prevState]();

        if (_friendStateEnter.ContainsKey(_currentState))
            _friendStateEnter[_currentState]();

        return;
    }

    base.ChangeState(state);
}

public Skill ActiveSkill(SkillType type, GameObject caster, Transform target)

Save Data

슬롯 기반 JSON 저장 구조

저장 데이터는 DataManager가 맡고, 현재 슬롯 이름으로 파일을 저장한다. 저장 책임을 한 곳에 모아 관리하는 구조다. 

public void LoadAllData()
{
    _slotData = new Dictionary<Slot, PlayerData>();

    for (int i = 0; i < (int)Slot.Max; i++)
    {
        Slot slot = (Slot)i;
        string fileName = slot.ToString();

        if (File.Exists(_path + fileName))
        {
            string JsonData = File.ReadAllText(_path + fileName);
            PlayerData data = JsonUtility.FromJson<PlayerData>(JsonData);
            _slotData.Add(slot, data);
        }
        else
        {
            _slotData.Add(slot, _newData);
        }
    }
}

public void SaveData(PlayerData data)
{
    string StrData = JsonUtility.ToJson(data);
    File.WriteAllText(_path + _currentSlot.ToString(), StrData);
}

Technical Strength

이 프로젝트에서 보여줄 수 있는 역량

상태 기반 로직 설계 능력

몬스터 행동을 단순 if 문 덩어리로 만들지 않고, 상태 전환 구조로 나누어 전투 흐름을 정리했습니다. 이 점은 AI 구조 설계 역량을 보여주기 좋습니다.

데이터와 실행 로직 분리

몬스터 수치, 스킬 실행, 저장 데이터가 각각 다른 책임으로 나뉘어 있습니다. 기능 추가 시 기존 코드 전체를 뜯지 않아도 되는 방향으로 설계한 점이 강점입니다.
