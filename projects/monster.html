프로젝트 설명

MonsterWorld

MonsterWorld는 몬스터를 포획하고, 포획한 몬스터를 아군으로 소환해 함께 전투하는 3D 액션 RPG 프로젝트입니다. 이 프로젝트에서는 단순 전투 구현보다 몬스터의 적/아군 상태 전환, 반복 생성되는 투사체와 이펙트 관리, 세이브 데이터 분리를 안정적으로 처리하는 구조 설계에 집중했습니다.

개발기간 2025.01 ~ 2025.03
장르 3D 액션 RPG
개발 형태 1인 개발
핵심 구조 상태머신 기반 AI / 오브젝트 풀 / 데이터 테이블

문제 상황

몬스터 포획 게임에서는 하나의 몬스터가 포획 전에는 적으로, 포획 후에는 아군으로 동작해야 합니다. 또한 총알, 화살, 스킬 이펙트처럼 전투 중 반복 생성되는 오브젝트가 많아질수록 생성/삭제 비용과 관리 복잡도가 증가할 수 있습니다.

해결 방향

이러한 문제를 해결하기 위해 다음과 같은 구조를 적용했습니다.

  • 제네릭 상태머신 기반 몬스터 AI 구조 적용
  • 적/아군 상태 딕셔너리 분리로 포획 전후 행동 전환 처리
  • Queue 기반 오브젝트 풀을 통한 반복 생성 비용 감소
  • JSON 저장 구조를 통한 세이브 데이터 분리

핵심 구현

  • 상태별 Enter / Update / Exit 함수를 Dictionary에 등록
  • Monster 클래스 내부에서 적군 상태와 아군 상태 흐름을 분리
  • 투사체와 이펙트를 풀에서 꺼내 사용하고 다시 반환
  • 슬롯별 플레이어 데이터를 JSON 파일로 저장 및 로드

결과

몬스터 포획 전후의 행동 전환을 별도 클래스 중복 없이 처리할 수 있었고, 전투 중 반복 생성되는 오브젝트를 재사용하여 구조적인 비용 증가를 줄였습니다.

이 프로젝트의 핵심 목표는 특정 성능 수치보다, 몬스터 AI·전투 오브젝트·저장 데이터를 각각 독립적으로 관리할 수 있는 유지보수 가능한 게임 구조를 설계하는 데 있습니다.

설계 포인트

제네릭 상태머신 기반 몬스터 AI

몬스터 행동을 Idle, Patrol, Chase, Attack, Return 상태로 구분하고, 각 상태의 Enter / Update / Exit 동작을 Dictionary에 등록하는 구조를 사용했습니다. 이를 통해 조건문이 계속 늘어나는 문제를 줄이고, 상태 추가 시 기존 흐름을 크게 변경하지 않아도 되는 구조를 만들었습니다.

적/아군 전환형 단일 몬스터 구조

포획 전에는 적 AI로, 포획 후에는 아군 AI로 동작해야 했기 때문에 몬스터 클래스를 두 벌로 나누지 않고 하나의 Monster 클래스 안에서 적군 상태와 아군 상태 흐름을 분리했습니다. 같은 데이터와 애니메이션 구조를 유지하면서 행동 규칙만 전환해 클래스 중복과 관리 비용을 줄였습니다.

오브젝트 풀 + 데이터 테이블 구조

투사체와 스킬 이펙트는 Queue 기반 오브젝트 풀로 재사용되도록 구성했습니다. 또한 플레이어 데이터와 포획 몬스터 정보는 JSON 저장 구조로 분리하여, 타이틀 씬과 인게임 씬에서 같은 세이브 데이터를 안정적으로 사용할 수 있도록 만들었습니다.

문제 해결

문제 1. 몬스터 AI 분기 증가

몬스터 상태가 늘어날수록 if / switch 기반 분기는 빠르게 복잡해질 수 있었습니다. 이를 해결하기 위해 상태별 함수를 Dictionary에 등록하는 제네릭 상태머신 구조를 사용했고, 상태 추가 시 기존 흐름을 크게 건드리지 않도록 정리했습니다.

문제 2. 포획 후 아군 전환 구조

포획 전과 후의 행동 패턴이 달라 별도 클래스로 나눌 수도 있었지만, 그렇게 하면 공통 데이터와 애니메이션 처리 중복이 커질 수 있었습니다. 그래서 하나의 Monster 클래스 내부에 적군용 상태와 아군용 상태 딕셔너리를 따로 두어 구조 중복을 줄이고 행동 전환만 분리했습니다.

문제 3. 전투 중 생성/삭제로 인한 비용 증가

투사체와 스킬 이펙트가 반복 생성되면 GC 부담과 프레임 저하가 발생할 수 있습니다. 이를 줄이기 위해 Queue 기반 오브젝트 풀을 직접 구현했고, 필요한 오브젝트를 새로 생성하기보다 풀에서 꺼내 사용한 뒤 다시 반환하도록 구성했습니다.

문제 4. 세이브 데이터 관리

플레이어 위치, 레벨, 포획 몬스터 정보는 여러 씬에서 함께 사용되어야 했기 때문에 씬 내부 오브젝트에만 의존하면 데이터 흐름이 복잡해질 수 있었습니다. 이를 해결하기 위해 슬롯별 데이터를 Dictionary로 관리하고, JsonUtility와 파일 입출력을 사용해 저장 데이터를 분리했습니다.

핵심 코드

제네릭 상태머신 구조

몬스터 AI의 상태 전환 로직을 공통화하기 위해 제네릭 상태머신을 만들었습니다. 상태별 Enter / Update / Exit 함수를 Dictionary에 등록하고, ChangeState에서 이전 상태 종료와 새 상태 진입을 일관된 순서로 처리하도록 구성했습니다. 

using System.Collections.Generic;
using UnityEngine;

public class StateMachineBase : MonoBehaviour
{
    protected delegate void StateFunction();

    protected Dictionary _stateEnter;
    protected Dictionary _stateUpdates;
    protected Dictionary _stateExit;

    public T _currentState;
    protected T _prevState;

    protected void InitStateMachine()
    {
        _stateEnter = new Dictionary();
        _stateUpdates = new Dictionary();
        _stateExit = new Dictionary();
    }

    protected virtual void ChangeState(T state)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_stateExit.ContainsKey(_prevState))
            _stateExit[_prevState]();

        if (_stateEnter.ContainsKey(_currentState))
            _stateEnter[_currentState]();
    }
}

적/아군 전환형 몬스터 AI

하나의 Monster 클래스 안에서 적군과 아군의 상태 흐름을 따로 구성해, 포획 전후 행동 패턴을 분리했습니다. 공통 데이터는 유지하면서 실제 행동 규칙만 전환하는 구조이기 때문에 클래스 중복을 줄이고 포획 시스템을 자연스럽게 연결할 수 있었습니다. 

protected Dictionary _friendStateEnter;
protected Dictionary _friendStateUpdate;
protected Dictionary _friendStateExit;

protected override void ChangeState(MonsterState state)
{
    _checkTime = 0;
    _stateTime = 0;

    if (_isFriend)
    {
        _prevState = _currentState;
        _currentState = state;

        if (_friendStateExit.ContainsKey(_prevState))
            _friendStateExit[_prevState]();

        if (_friendStateEnter.ContainsKey(_currentState))
            _friendStateEnter[_currentState]();

        return;
    }
    base.ChangeState(state);
}

JSON 기반 세이브 데이터 관리

세이브 슬롯을 Dictionary로 관리하고, 실제 저장은 JsonUtility와 파일 입출력으로 처리했습니다. 플레이어 위치, 레벨, 포획 몬스터 정보를 PlayerData로 보관해 타이틀 씬과 인게임 씬에서 공통으로 사용할 수 있는 형태로 구성했습니다. 

public void LoadAllData()
{
    _slotData = new Dictionary();

    int slotCount = (int)Slot.Max;

    for (int i = 0; i < slotCount; i++)
    {
        Slot slot = (Slot)i;
        string fileName = (slot).ToString();

        if (File.Exists(_path + fileName))
        {
            string JsonData = File.ReadAllText(_path + fileName);
            PlayerData data = JsonUtility.FromJson(JsonData);
            _slotData.Add(slot, data);
        }
        else
            _slotData.Add(slot, _newData);
    }
}

배운 점

이 프로젝트를 통해 AI를 조건문 중심으로 구현하는 것보다, 상태 단위로 책임을 분리하는 구조가 유지보수에 유리하다는 점을 체감했습니다. 특히 포획 전후처럼 행동 패턴이 크게 바뀌는 경우에도 공통 데이터 구조를 유지하고 행동 흐름만 분리하면 중복을 줄일 수 있다는 점을 배웠습니다.

개선 방향

앞으로는 몬스터 종류별 고유 패턴을 ScriptableObject 또는 별도 행동 트리 구조로 분리해, 코드 수정 없이 몬스터 행동을 확장할 수 있는 방향으로 개선할 계획입니다. 또한 현재의 매니저 중심 구조를 더 느슨한 의존성 구조로 바꿔 테스트와 유지보수를 쉽게 만드는 방향도 검토하고 있습니다.
