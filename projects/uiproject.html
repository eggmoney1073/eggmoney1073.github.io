프로젝트 설명

UI Project

게임 플레이 이전 단계의 사용자 경험을 설계하고, 이를 실제 게임 흐름으로 연결하는 UI 시스템 프로젝트입니다. 타이틀 → 다운로드 확인 → 로딩 → 메인 메뉴 → 게임 진입까지의 과정을 단순한 화면 전환이 아니라 하나의 UX 파이프라인으로 정리하는 데 집중했습니다.

개발기간 약 3일
장르 UI / UX 시스템 설계
개발 형태 1인 개발
핵심 구조 씬 기반 UX 흐름 / 입력 분리 / 팝업 공통 구조

이 프로젝트의 목표는 UI를 단순한 배치 작업이 아니라 게임 진입 전후의 흐름을 제어하는 시스템으로 설계해보는 것이었습니다. 다운로드 여부 확인, 로딩 상태 표시, 메뉴 진입, 입력 상태 분리 같은 기능을 단계별로 나누어, 실제 게임 프로젝트에 바로 연결할 수 있는 형태의 UI 구조를 만드는 데 초점을 맞췄습니다.

설계 포인트

씬 단위 UX 흐름 분리

타이틀, 다운로드, 로딩, 메인 메뉴, 게임 진입 단계를 각각 독립된 흐름으로 나누어 설계했습니다. 단순히 버튼 클릭으로 화면만 바꾸는 것이 아니라, 사용자가 자연스럽게 다음 단계로 이동하도록 UX 관점에서 구조를 정리했습니다.

UI 입력과 게임 입력 분리

게임 플레이 입력과 UI 조작 입력이 섞이면 유지보수와 디버깅이 어려워질 수 있기 때문에, Input System 기반으로 상태를 나누고 상황에 따라 입력 영역을 분리하는 방향으로 구조를 잡았습니다.

팝업 공통 구조 설계

팝업은 기능이 늘어날수록 중복 코드가 빠르게 생기기 때문에, BasePopup 기반 공통 구조와 중앙 관리 방식을 기준으로 확장 가능한 형태를 의식해 설계했습니다.

문제 해결

문제 1. UI를 화면 단위로만 보면 흐름이 끊김

처음에는 각 화면을 개별 UI로만 보게 되어 다운로드, 로딩, 진입 흐름이 따로 노는 문제가 있었습니다. 이를 UX 흐름 단위로 다시 정리하면서 단계별 역할을 분리하는 방식으로 구조를 개선했습니다.

문제 2. UI 입력과 게임 입력 충돌 가능성

메뉴 조작과 게임 조작이 동시에 열려 있으면 예기치 않은 입력 충돌이 발생할 수 있습니다. 이를 막기 위해 상황에 따라 어떤 입력이 활성화되는지 명확히 나누는 구조를 우선적으로 설계했습니다.

문제 3. 팝업 확장 시 중복 코드 증가

팝업이 추가될 때마다 열기/닫기/초기화 코드가 반복될 가능성이 있어, 공통 부모 구조와 매니저 중심 관리 방식을 기준으로 확장성을 확보하려고 했습니다.

핵심 코드

UX 흐름 분리 구조

이 프로젝트의 핵심은 여러 화면을 따로 만드는 것이 아니라, 게임 시작 전 과정을 하나의 UX 파이프라인으로 나누는 구조에 있습니다. 포트폴리오에서는 아래처럼 단계가 분리된 구조 자체를 핵심 코드 관점으로 보여주는 것이 맞습니다. 

TitleScene
→ DownloadCheck
→ LoadingScene
→ MainMenu
→ GameScene

입력 상태 분리 구조

메뉴 조작과 플레이 조작이 동시에 열려 있으면 충돌 가능성이 커지기 때문에, 어떤 상황에서 어떤 입력을 활성화할지 분리하는 구조가 핵심입니다. 

if (currentState == UIState.Menu)
{
    EnableUIInput();
    DisableGameInput();
}
else if (currentState == UIState.InGame)
{
    DisableUIInput();
    EnableGameInput();
}

팝업 공통 구조

팝업이 늘어날수록 열기, 닫기, 초기화 코드가 반복되기 때문에, 공통 부모와 중앙 관리 구조로 묶는 방식이 이 프로젝트의 핵심 설계 포인트입니다. 

public abstract class BasePopup : MonoBehaviour
{
    public virtual void Open() { gameObject.SetActive(true); }
    public virtual void Close() { gameObject.SetActive(false); }
    public virtual void Initialize() { }
}

public class PopupManager : MonoBehaviour
{
    public void Show(BasePopup popup)
    {
        popup.Initialize();
        popup.Open();
    }
}

배운 점

이 프로젝트를 통해 UI는 단순히 화면을 예쁘게 구성하는 작업이 아니라, 사용자가 게임에 진입하기까지의 흐름 전체를 설계하는 일이라는 점을 다시 정리할 수 있었습니다. 특히 다운로드·로딩·입력 상태처럼 플레이 이전 단계의 경험도 시스템적으로 다뤄야 한다는 점이 중요했습니다.

개선 방향

앞으로는 이 구조를 실제 코드 단위 상태머신이나 UI 상태 컨트롤러와 더 명확하게 연결하고, 팝업과 로딩 흐름을 데이터 기반으로 관리할 수 있는 구조로 확장해볼 계획입니다. 또한 프로젝트 규모가 커질수록 UI 테스트와 상태 추적이 쉬운 형태로 일반화하는 것도 개선 방향입니다.
