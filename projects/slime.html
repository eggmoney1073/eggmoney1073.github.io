Project Detail

Slime Defense

Unity ECS/DOTS 기반으로 다수의 적과 투사체를 효율적으로 처리하기 위해 전투 로직을 시스템 파이프라인으로 설계한 1인 디펜스 프로젝트입니다. Grid 기반 충돌 최적화와 LiveTag 기반 생명주기 분리를 적용해, 대량 엔티티 처리 상황에서도 안정적으로 동작하는 구조를 구현했습니다.

Overview

프로젝트 개요

장르

세로형 디펜스

5분 생존형 전투 루프를 기반으로 한 모바일 디펜스 프로젝트입니다.

개발 형태

1인 개발

기획, 설계, 구현, UI 연결, 리소스 관리 구조까지 전반을 직접 구성했습니다.

핵심 구조

ECS 전투 파이프라인

Spawn → Move → Judgement → Calculate → Destroy 단계로 전투 흐름을 분리했습니다.

핵심 기술

Grid + LiveTag

충돌 탐색 비용 최적화와 안전한 엔티티 삭제 구조를 함께 설계했습니다.

프로젝트 목표

이 프로젝트의 목표는 단순히 게임을 완성하는 것이 아니라, 다수의 적과 투사체가 동시에 존재하는 상황에서 전투 로직을 어떻게 안정적으로 설계할지 검증하는 것이었습니다. 이를 위해 전투 흐름을 ECS 시스템 그룹으로 단계화했고, 충돌 판정과 삭제 처리에서 발생할 수 있는 성능 문제와 타이밍 충돌 문제를 구조적으로 해결하는 데 집중했습니다.

기술 스택

Unity 6000 / C# / ECS(DOTS) / Burst / Addressables / Input System / URP

Gameplay

게임 플레이

전투, 레벨업 선택, 성장 흐름을 중심으로 세로형 플레이 화면을 구성했습니다.

핵심 플레이 루프

적 처치 → 경험치 획득 → 레벨업 선택 → 무기 및 능력 강화 → 추가 웨이브 대응의 반복 구조로 설계했습니다. 짧은 플레이 시간 안에서도 성장 선택과 전투 밀도를 동시에 느낄 수 있도록 구성했습니다.

Architecture

설계 포인트

ECS 전투 파이프라인

전투 로직을 하나의 거대한 Update에서 처리하지 않고, 관리, 생성, 이동, 판정, 계산, 삭제 단계로 분리했습니다. 이 구조를 통해 각 시스템이 담당하는 책임을 명확하게 나눌 수 있었고, 실행 순서를 고정해 디버깅과 유지보수가 쉬운 흐름을 만들었습니다.

Grid 기반 충돌 구조

디펜스 게임에서는 적과 투사체 수가 증가할수록 충돌 비용이 급격히 커집니다. 이를 해결하기 위해 적 위치를 Grid에 배치하고, 투사체는 현재 위치 기준 주변 셀만 탐색하도록 설계해 충돌 후보군을 제한했습니다.

LiveTag 기반 생명주기 분리

판정 중간에 엔티티를 즉시 제거하지 않고, 먼저 비활성 상태로 전환한 뒤 마지막 단계에서 일괄 삭제하도록 구성했습니다. 이 방식으로 같은 프레임 내 참조 충돌 가능성을 줄이고, 전투 파이프라인 전체를 더 안전하게 유지했습니다.

Problem Solving

문제 해결

문제 1. 충돌 판정 비용 증가

적과 투사체 수가 늘어날수록 모든 엔티티를 직접 비교하는 구조는 빠르게 비효율적이 되었습니다. 이를 해결하기 위해 Grid 기반 공간 분할을 도입했고, 각 투사체가 자신의 주변 3x3 셀만 탐색하도록 변경하여 충돌 탐색 범위를 줄였습니다.

문제 2. 중복 피격 처리

투사체가 같은 적을 한 프레임 안에서 여러 번 처리할 가능성이 있어, 피격 기록을 별도로 관리할 필요가 있었습니다. 이를 위해 HitEnemyBufferElement를 두어 이미 타격한 적을 기록하고, Damaged 버퍼에 피해를 누적한 뒤 계산 단계에서 일괄 처리하도록 구성했습니다.

문제 3. 엔티티 삭제 타이밍 충돌

충돌 판정 중간에 엔티티를 바로 Destroy하면 같은 프레임 내 다른 시스템과 충돌할 수 있습니다. 그래서 죽음 판정과 실제 삭제를 분리하고, LiveTag를 비활성화한 후 Destroy 단계에서 EntityCommandBuffer로 일괄 제거하도록 설계했습니다.

System Flow

ECS 전투 파이프라인

전투 로직을 하나의 흐름으로 처리하지 않고, 스폰 → 이동 → 판정 → 계산 → 삭제 단계로 명확히 분리한 구조입니다.

왜 이렇게 설계했는가?
ECS에서는 시스템 실행 순서가 보장되지 않으면 충돌 판정이나 삭제 타이밍에서 버그가 발생할 수 있습니다. 이를 해결하기 위해 UpdateAfter를 사용해 단계별 실행 순서를 고정했고, 각 시스템이 하나의 책임만 가지도록 분리했습니다.

Collision

Grid 기반 충돌 최적화

모든 적과 투사체를 직접 비교하지 않고, Grid 기반 공간 분할을 통해 충돌 후보군을 제한하는 구조입니다.

왜 이렇게 설계했는가?
디펜스 게임에서는 적과 투사체 수가 늘어날수록 충돌 검사 비용이 급격히 증가합니다. 이를 해결하기 위해 위치를 Grid에 저장하고, 투사체는 자신의 주변 3x3 셀만 탐색하도록 설계했습니다.
핵심 포인트
GridBuild로 위치 저장 / 3x3 셀 탐색 / HitBuffer로 중복 피격 방지 / Damaged 버퍼 누적 
using Unity.Burst;
using Unity.Collections;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
[WithAll(typeof(ProjectileTag))]
public partial struct Collision_Job : IJobEntity
{
    public int _gridWidth;
    public float _cellSize;

    [ReadOnly]
    public NativeParallelMultiHashMap<int, Entity> _gridMap;

    [ReadOnly]
    public ComponentLookup<LocalTransform> _transformLookup;
    [ReadOnly]
    public ComponentLookup<Collision> _radiusLookup;

    [NativeDisableParallelForRestriction]
    public BufferLookup<HitEnemyBufferElement> _hitBufferLookup;

    [NativeDisableParallelForRestriction]
    public BufferLookup<Damaged> _damagedBufferLookup;

    public void Execute(Entity projectileEntity, in LocalTransform transform, in Collision collider, in ProjectileDamage projectileDamage)
    {
        float3 position = transform.Position;

        int centerX = (int)math.floor(position.x / _cellSize) + (_gridWidth / 2);
        int centerY = (int)math.floor(position.z / _cellSize) + (_gridWidth / 2);

        int offsetY = -1;
        while (offsetY <= 1)
        {
            int offsetX = -1;
            while (offsetX <= 1)
            {
                int axisX = centerX + offsetX;
                int axisY = centerY + offsetY;
                int index = axisX + _gridWidth * axisY;

                NativeParallelMultiHashMapIterator<int> iterator;
                Entity other;

                bool found = _gridMap.TryGetFirstValue(index, out other, out iterator);

                if (found)
                {
                    do
                    {
                        float distanceSq = math.lengthsq(position - _transformLookup[other].Position);
                        float sumRadius = collider.radius + _radiusLookup[other].radius;

                        if (distanceSq <= sumRadius * sumRadius)
                        {
                            DynamicBuffer<HitEnemyBufferElement> hitBuffer = _hitBufferLookup[projectileEntity];
                            DynamicBuffer<Damaged> damagedBuffer = _damagedBufferLookup[other];

                            hitBuffer.Add(new HitEnemyBufferElement { enemy = other });
                            damagedBuffer.Add(new Damaged { damage = projectileDamage.damage });
                        }
                    }
                    while (_gridMap.TryGetNextValue(out other, ref iterator));
                }

                offsetX++;
            }

            offsetY++;
        }
    }
}

Lifecycle

LiveTag 기반 생명주기 관리

엔티티를 즉시 제거하지 않고, 먼저 비활성화한 뒤 마지막 단계에서 정리하는 구조입니다.

왜 이렇게 설계했는가?
충돌 판정 중간에 엔티티를 바로 삭제하면 같은 프레임 내 다른 시스템과 참조 충돌이 발생할 수 있습니다. 이를 방지하기 위해 죽음 판정과 실제 삭제를 분리했고, LiveTag 비활성화 후 Destroy 단계에서 EntityCommandBuffer로 일괄 제거하도록 설계했습니다.
핵심 포인트
상태 전환과 실제 제거 분리 / WithDisabled 조건 조회 / 안전한 일괄 삭제 구조 
using Unity.Entities;

public struct LiveTag : IComponentData, IEnableableComponent { }

[WithAll(typeof(EnemyTag))]
partial struct EnemyDead_System : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        foreach ((RefRO<EnemyHealth> enemyHealth, EnabledRefRW<LiveTag> liveTag, Entity entity)
            in SystemAPI.Query<RefRO<EnemyHealth>, EnabledRefRW<LiveTag>>().WithEntityAccess())
        {
            if (enemyHealth.ValueRO.health <= 0)
            {
                liveTag.ValueRW = false;

                RefRW<GameProgress> progress = SystemAPI.GetSingletonRW<GameProgress>();
                progress.ValueRW.kill += 1;
            }
        }
    }
}

public partial struct EntityDestroy_Job : IJobEntity
{
    public EntityCommandBuffer.ParallelWriter ecbWriter;
}

배운 점

이 프로젝트를 통해 ECS는 단순히 성능을 위한 기술이 아니라, 데이터 흐름과 실행 순서를 명확히 설계해야 효과를 발휘하는 구조라는 점을 배웠습니다. 특히 충돌 처리와 삭제 처리처럼 프레임 내 타이밍이 중요한 영역에서는 문제를 기능 단위가 아니라 시스템 흐름 단위로 바라보는 사고가 중요하다는 점을 확인했습니다.

개선 방향

앞으로는 Grid 크기와 셀 크기를 데이터 기반으로 분리하고, 무기 및 적 패턴의 확장성을 높일 수 있도록 전투 데이터를 더 세분화할 계획입니다. 또한 UI와 게임 진행 상태 연결 구조를 더 일반화해, 실제 서비스형 프로젝트에 가까운 형태로 확장할 예정입니다.
